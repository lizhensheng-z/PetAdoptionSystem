## API 通用规范

### 基本约定

| 项目 | 说明 |
|-----|------|
| **基础路径** | `/api` |
| **版本** | `v1`（可选：`/api/v1/...` 或统一使用 `/api/...`） |
| **数据格式** | JSON |
| **字符编码** | UTF-8 |
| **日期格式** | ISO 8601（`YYYY-MM-DDTHH:mm:ss`） |

### 统一响应格式

**成功响应（HTTP 200）：**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    // 业务数据
  },
  "timestamp": "2024-01-20T10:30:00Z"
}
```

**分页响应：**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [/* items */],
    "pageNo": 1,
    "pageSize": 10,
    "total": 100,
    "totalPages": 10
  },
  "timestamp": "2024-01-20T10:30:00Z"
}
```

**错误响应（HTTP 4xx/5xx）：**
```json
{
  "code": 1001,
  "message": "用户不存在或密码错误",
  "timestamp": "2024-01-20T10:30:00Z"
}
```

### 错误码定义

| 错误码 | HTTP 状态 | 说明 |
|-------|---------|------|
| 0 | 200 | 成功 |
| 1001 | 401 | 认证失败（用户名/密码错误、token过期等） |
| 1002 | 403 | 权限不足 |
| 1003 | 400 | 参数验证失败 |
| 1004 | 404 | 资源不存在 |
| 1005 | 409 | 业务冲突（如重复申请、重复收藏） |
| 1006 | 429 | 请求过于频繁（限流） |
| 1007 | 500 | 服务器错误 |

### 权限声明

所有接口中将标注所需权限：
- `无需认证`：游客可访问
- `需认证`：登录用户可访问
- `权限code`：需要特定权限（如 `pet:create`）

### 请求头要求

```
Authorization: Bearer <access_token>
Content-Type: application/json
```

## 认证与授权接口

### 1. 用户注册

**接口：** `POST /api/auth/register`

**权限：** 无需认证

**请求体：**
```json
{
  "username": "john_doe",
  "password": "securePassword123!",
  "phone": "13800000001",
  "email": "john@example.com",
  "role": "USER"
}
```

**参数说明：**
| 参数 | 类型 | 必填 | 说明 |
|-----|------|-----|------|
| username | string | Y | 用户名，3-64字符，唯一 |
| password | string | Y | 密码，8字符以上，需包含大小写+数字 |
| phone | string | N | 手机号，11位，唯一 |
| email | string | N | 邮箱 |
| role | enum | Y | 角色：`USER`（领养人） / `ORG`（机构） |

**成功响应（HTTP 201）：**
```json
{
  "code": 0,
  "message": "注册成功",
  "data": {
    "userId": 1001,
    "username": "john_doe",
    "role": "USER"
  }
}
```

**失败响应示例：**
```json
{
  "code": 1003,
  "message": "用户名已存在"
}
```

**业务规则：**
- 用户名、手机号、邮箱（如提供）必须唯一
- 密码需加密存储（bcrypt或argon2）
- role=ORG 时，后续需补充 `org_profile`

### 2. 用户登录

**接口：** `POST /api/auth/login`

**权限：** 无需认证

**请求体：**
```json
{
  "username": "john_doe",
  "password": "securePassword123!",
  "type": "password"
}
```

**参数说明：**
| 参数 | 类型 | 必填 | 说明 |
|-----|------|-----|------|
| username | string | Y | 用户名或邮箱或手机号 |
| password | string | Y | 密码 |
| type | enum | N | 登录方式，默认 `password`；可扩展微信/支付宝 |

**成功响应（HTTP 200）：**
```json
{
  "code": 0,
  "message": "登录成功",
  "data": {
    "accessToken": "eyJhbGc...",
    "refreshToken": "eyJhbGc...",
    "expiresIn": 3600,
    "tokenType": "Bearer",
    "user": {
      "userId": 1001,
      "username": "john_doe",
      "role": "USER",
      "avatar": "https://...",
      "phone": "13800000001",
      "email": "john@example.com"
    }
  }
}
```

**业务规则：**
- JWT token 有效期建议：accessToken 1h，refreshToken 7d
- 每次登录更新 `sys_user.last_login_time`
- 账户被封禁（status=BANNED）拒绝登录

| 参数 | 类型 | 必填 | 说明 |
|-----|------|-----|------|
| pageNo | int | N | 页码，默认1 |
| pageSize | int | N | 每页条数，默认10，最大100 |
| species | enum | N | 物种：CAT、DOG、OTHER |
| breed | string | N | 品种（模糊搜索） |
| gender | enum | N | 性别：MALE、FEMALE、UNKNOWN |

| tags | string | N | 标签ID列表，逗号分隔 |

| lat | decimal | N | 用户纬度 |

| sortBy | enum | N | 排序字段：`publishedTime`、`distance`、`match`，默认 `publishedTime` |

### 9. 机构创建宠物档案

**接口：** `POST /api/org/pets`

**权限：** 需认证 + `pet:create`

**请求体：**
```json
{
  "name": "小花",
  "species": "CAT",
  "breed": "英短",
  "gender": "FEMALE",
  "ageMonth": 24,
  "size": "M",
  "color": "灰色",
  "sterilized": true,
  "vaccinated": true,
  "dewormed": true,
  "healthDesc": "身体健康，精神状态良好",
  "personalityDesc": "性格温顺，亲人",
  "adoptRequirements": "需要有爱心和耐心",
  "tagIds": [101, 102, 103],
  "lng": 116.4,
  "lat": 39.9
}
```

**参数说明：**
| 参数 | 类型 | 必填 | 说明 |
|-----|------|-----|------|
| name | string | N | 宠物名字/昵称，最多64字符 |
| species | enum | Y | 物种：CAT、DOG、OTHER |
| breed | string | N | 品种，最多64字符 |
| gender | enum | N | 性别：MALE、FEMALE、UNKNOWN |
| ageMonth | int | N | 年龄（月） |
| size | enum | N | 体型：S、M、L |
| color | string | N | 毛色/颜色，最多32字符 |
| sterilized | boolean | N | 是否绝育 |
| vaccinated | boolean | N | 是否疫苗 |
| dewormed | boolean | N | 是否驱虫 |
| healthDesc | string | N | 健康描述，最多1000字 |
| personalityDesc | string | N | 性格描述，最多1000字 |
| adoptRequirements | string | N | 领养要求，最多1000字 |
| tagIds | array | N | 标签ID数组 |
| lng | decimal | N | 经度（不填则继承机构坐标） |
| lat | decimal | N | 纬度 |

**成功响应（HTTP 201）：**
```json
{
  "code": 0,
  "message": "宠物档案创建成功",
  "data": {
    "id": 2001,
    "status": "DRAFT",
    "auditStatus": "NONE",
    "createTime": "2024-01-20T10:00:00Z"
  }
}
```

**业务规则：**
- 新建宠物状态为 DRAFT，auditStatus 为 NONE
- org_user_id 从 JWT token 中获取，确保只能创建自己机构的宠物
- tagIds 数组在请求时验证其存在性
- lng/lat 不填则继承 org_profile 坐标

### 10. 机构修改宠物档案

**接口：** `PUT /api/org/pets/{petId}`

**权限：** 需认证 + `pet:update`

**路径参数：**
| 参数 | 类型 | 说明 |
|-----|------|------|
| petId | long | 宠物ID |

**请求体：**（同创建，但所有字段可选）
```json
{
  "name": "小花儿",
  "healthDesc": "更新的健康信息"
}
```

**成功响应：**
```json
{
  "code": 0,
  "message": "更新成功"
}
```

**业务规则：**
- 只能修改 status=DRAFT 或 REJECTED 的宠物
- 若宠物已 PUBLISHED，禁止修改核心字段（可通过新建版本或允许修改某些非核心字段如图片）
- 验证操作者是否为宠物所属机构
- 修改后 auditStatus 重置为 NONE

### 11. 上传宠物媒体

**接口：** `POST /api/org/pets/{petId}/media`

**权限：** 需认证 + `pet:update`

**路径参数：**
| 参数 | 类型 | 说明 |
|-----|------|------|
| petId | long | 宠物ID |

**请求体（multipart/form-data）：**
```
file: <binary file>
mediaType: IMAGE
sort: 1
```

| 参数 | 类型 | 必填 | 说明 |
|-----|------|-----|------|
| file | file | Y | 图/视频文件，IMAGE 类型最大 10MB，VIDEO 最大 100MB |
| mediaType | enum | Y | IMAGE 或 VIDEO |
| sort | int | N | 排序号，默认自动递增 |

**成功响应（HTTP 201）：**
```json
{
  "code": 0,
  "message": "媒体上传成功",
  "data": {
    "id": 5001,
    "petId": 2001,
    "url": "https://example.com/pet-2001-1.jpg",
    "mediaType": "IMAGE",
    "sort": 1
  }
}
```

**业务规则：**
- 调用对象存储服务（COS/OSS）上传文件
- 返回 url 为外网可访问地址
- 单个宠物最多上传 30 个媒体文件

### 22. 机构流转申请状态

**接口：** `POST /api/org/applications/{applicationId}/status`

**权限：** 需认证 + `app:org:status`

**请求体：**
```json
{
  "toStatus": "INTERVIEW",
  "interviewTime": "2024-01-25T14:00:00Z",
  "interviewLocation": "北京市朝阳区xxx",
  "remark": "初审通过，请于指定时间参加面谈"
}
```

或

```json
{
  "toStatus": "APPROVED",
  "remark": "很好的申请人，已通过，祝小花和新主人幸福"
}
```

或

```json
{
  "toStatus": "REJECTED",
  "rejectReason": "家庭条件不符合领养要求",
  "remark": "感谢您的申请，但贵家庭暂不符合我们的要求"
}
```

**参数说明：**
| 参数 | 类型 | 必填 | 说明 |
|-----|------|-----|------|
| toStatus | enum | Y | 目标状态：UNDER_REVIEW、INTERVIEW、HOME_VISIT、APPROVED、REJECTED、CANCELLED |
| rejectReason | string | N | 拒绝原因（toStatus=REJECTED 时必填） |
| remark | string | N | 备注/通知文案 |
| interviewTime | datetime | N | 面谈时间（toStatus=INTERVIEW 时推荐提供） |
| interviewLocation | string | N | 面谈地点 |

**成功响应：**
```json
{
  "code": 0,
  "message": "状态更新成功",
  "data": {
    "applicationId": 6001,
    "fromStatus": "SUBMITTED",
    "toStatus": "INTERVIEW",
    "updateTime": "2024-01-20T11:00:00Z"
  }
}
```

**业务规则（状态机）：**
```
状态转移允许关系：
SUBMITTED -> UNDER_REVIEW / REJECTED / CANCELLED
UNDER_REVIEW -> INTERVIEW / REJECTED / CANCELLED
INTERVIEW -> HOME_VISIT / APPROVED / REJECTED / CANCELLED
HOME_VISIT -> APPROVED / REJECTED / CANCELLED
APPROVED -> (终止)
REJECTED -> (终止)
CANCELLED -> (终止)
```

- 状态更新时创建 adoption_flow_log
- 若 toStatus=APPROVED，则更新宠物 status=ADOPTED，并关闭其他申请（自动拒绝）
- 异步发送通知给申请人
- 若 toStatus=APPROVED，创建打卡提醒（定时任务）

**业务规则：**
- 同一用户对同一宠物同一天的相同行为只记录一次（或合并计数）
- 根据 behaviorType 赋予权重：VIEW=1, FAVORITE=3, APPLY=5, SHARE=2, CHECKIN=2
- 异步写入数据库，不阻塞主流程
- 数据用于协同过滤推荐训练

## 附录：特殊业务流程说明

### A. 信用分计算逻辑

```
基础规则：
- 每个行为产生一条 credit_log 记录
- 定期（如每天夜间）聚合结算，更新 credit_account

具体计算：

1. 打卡行为 (CHECKIN):
   base_score = sys_config.credit.checkin.base (默认2分)
   if media_count > 0:
       score += sys_config.credit.checkin.mediaBonus (默认1分)
   if content_length > 30:
       score += sys_config.credit.checkin.textBonus (默认1分)
   max_daily = sys_config.credit.checkin.dailyMax (默认1次)
   
   示例：有文字+图片的打卡 = 2 + 1 + 1 = 4分

2. 违规行为 (VIOLATION):
   penalty = sys_config.credit.violation.penalty (默认-20分)
   可根据违规严重程度调整倍数
   示例：虐待宠物 = -20分

3. 申请通过 (APPLICATION_APPROVED):
   已采纳的方案：在打卡时计分，申请本身不计分
   可选方案：申请通过时 +5分

4. 等级计算 (LEVEL):
   if score < 25: level = 0 (新注册)
   else if score < 50: level = 1 (铜牌)
   else if score < 100: level = 2 (银牌)
   else if score < 150: level = 3 (金牌)
   else: level = 4 (铂金)
```

### B. 推荐分计算逻辑

```
最终推荐分 = α·match_score + β·cf_score + γ·distance_score + δ·freshness_score

α = sys_config.recommend.alpha (默认0.55) - 偏好匹配权重
β = sys_config.recommend.beta (默认0.20) - 协同过滤权重
γ = sys_config.recommend.gamma (默认0.20) - 距离权重
δ = sys_config.recommend.delta (默认0.05) - 新鲜度权重

1. match_score (0-100):
   base = 0
   if pet.species == user.preference.species: base += 30
   if pet.size in user.preference.sizes: base += 15
   if pet.age in user.preference.ageRange: base += 15
   for tag in pet.tags:
       if tag in user.preference.tags: base += 8 (最多3个标签)
   if user.preference.requiredHealth:
       matched_count = count(pet健康特性 & user偏好)
       base += 10 * (matched_count / required_count)
   match_score = min(base, 100)

2. cf_score (0-100):
   根据用户行为相似度推荐得分
   - 相似用户收藏/申请过的宠物
   - 或相似宠物（与用户已交互宠物相似的）
   cf_score = 相似度 * 100

3. distance_score (0-20):
   if distance <= 5km: score = 20
   else if distance <= 20km: score = 20 * (1 - (distance-5)/15)
   else if distance <= maxDistance: score = 20 * (1 - distance/maxDistance)
   else: score = 0
   maxDistance = sys_config.recommend.maxDistanceKm (默认50)

4. freshness_score (0-10):
   days_since_publish = now - pet.publishedTime
   if days_since_publish <= 7: score = 10
   else if days_since_publish <= 30: score = 10 * (1 - (days-7)/23)
   else: score = 0
```

### C. 申请状态机

```
状态转移图：

SUBMITTED (用户提交)
    ↓ (机构审核)
UNDER_REVIEW (机构初审)
    ├→ REJECTED (拒绝)
    └→ INTERVIEW (约面谈)
         ├→ REJECTED (拒绝)
         └→ HOME_VISIT (家访，可选)
             ├→ REJECTED (拒绝)
             └→ APPROVED (通过)
                  ↓
                  宠物状态变更为 ADOPTED
                  其他申请自动变更为 REJECTED

CANCELLED (任何阶段可撤回)

关键业务规则：
- 同一宠物最多 N 个申请进行中（configurable）
- 申请通过后，宠物自动标记为 ADOPTED
- 其他该宠物的进行中申请自动拒绝（并通知）
- 每次状态变更记录流转日志
- 每次状态变更异步发送通知
```

### D. 宠物生命周期

```
生命周期状态：

DRAFT (草稿)
    ↓ (提交审核)
PENDING_AUDIT (待审核)
    ├→ REJECTED (驳回，返回DRAFT)
    └→ PUBLISHED (发布通过)
         ├→ APPLYING (有申请进行中，自动标记)
         └→ ADOPTED (领养通过)
         └→ REMOVED (下架/删除)

业务规则：
- DRAFT 状态机构可修改宠物信息
- PENDING_AUDIT 状态禁止修改
- PUBLISHED 状态有人申请时自动变更为 APPLYING
- ADOPTED 或 REMOVED 为终止状态
- 逻辑删除不真实删除数据（deleted=1）
```

### E. 打卡与回访规则

```
自动打卡提醒流程：

1. 申请通过（APPROVED）时：
   - 生成打卡任务：7天、30天、90天
   - 存储在定时任务表或消息队列

2. 定时任务执行（如凌晨2点）：
   - 扫描待提醒的打卡任务
   - 计算是否超期（距离adoptedTime的天数）
   - 若未打卡：
     if 超期3天内：level=warning, 发送提醒通知
     if 超期3天以上：level=alert, 发送催促通知 + 机构提醒
     if 超期30天：level=violation, 扣信用分（-5分），标记用户为风险

3. 机构回访：
   - 可查看领养人打卡情况
   - 若发现异常（无理由未打卡/虐待迹象）：
     → 标记黑名单（拒绝后续申请）
     → 上报管理员审查
     → 可触发违规扣分
```

### F. 权限边界检查清单

```
宠物操作权限：
✓ 机构只能修改/删除自己发布的宠物
✓ 管理员可查看所有宠物的审核状态
✓ 用户只能查看已发布的宠物

申请操作权限：
✓ 用户只能查看/撤回自己的申请
✓ 机构只能查看/处理自己宠物的申请
✓ 管理员无需审查申请（仅机构）

个人资料权限：
✓ 用户只能修改自己的资料和偏好
✓ 机构只能修改自己的机构资料
✓ 管理员可修改所有用户账户状态

管理员权限：
✓ 仅 role=ADMIN 的用户可访问管理接口
✓ 管理员操作需记录审计日志
```

2. 或先按经纬度范围过滤，再精确计算距离
3. 或调用专业 LBS 服务
```

### 异步处理

**宜异步的操作：**
1. 行为埋点记录
2. 推荐分计算（离线）
3. 信用分结算（定时）
4. 邮件/短信通知
5. 审计日志记录
6. 文件病毒扫描、图片审核
7. 打卡提醒任务

**使用消息队列（RabbitMQ/Kafka）或定时任务框架（Quartz/XXL-Job）**

## 附录：安全性检查清单

### 认证与授权
- [ ] 所有敏感接口均需认证（Authorization header）
- [ ] JWT token 有效期不超过 1 小时，refreshToken 不超过 7 天
- [ ] 登出时将 token 加入黑名单（Redis）
- [ ] 权限验证在 controller 或 AOP 层统一处理
- [ ] 禁止直接在 URL 中传递敏感信息（如密码）

### 输入验证
- [ ] 参数长度/格式校验（JSR-380 注解）
- [ ] SQL 注入防护（使用 PreparedStatement，ORM 框架）
- [ ] XSS 防护（输出转义，CSP 策略）
- [ ] CSRF 防护（token 验证，SameSite Cookie）
- [ ] 文件上传类型/大小限制，病毒扫描

### 数据保护
- [ ] 密码使用 bcrypt/argon2 加密，不存储明文
- [ ] 敏感字段加密（如身份证、银行卡信息）
- [ ] 逻辑删除不真实删除数据
- [ ] 定期备份数据库
- [ ] 敏感信息接口返回时脱敏（如手机号、邮箱）

### API 安全
- [ ] 速率限制（RateLimit）防止滥用
- [ ] 接口版本管理，过期接口逐步下线
- [ ] 错误响应不泄露内部实现细节
- [ ] 记录所有关键操作的审计日志

### 网络安全
- [ ] HTTPS 传输，禁用 HTTP
- [ ] CORS 配置严格，仅允许必要的域名
- [ ] 防火墙规则，限制数据库直接访问
- [ ] DDoS 防护（如 CDN、WAF）

### 第三方服务
- [ ] API key/secret 不硬编码，使用环境变量或配置中心
- [ ] 第三方服务调用有超时机制
- [ ] 定期审查第三方权限和数据
