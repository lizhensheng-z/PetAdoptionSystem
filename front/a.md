# 揭秘 Agent 的高级面纱：一个初级 Java 程序员从困惑到“能做旅游规划助手”的全记录

我一开始的困惑很朴素：  
**LLM 不就是会聊天吗？为什么大家突然开始聊 Agent、RAG、MCP、Workflow？**  
听起来像一堆高端名词，但又隐约在讲同一件事：让 AI **不止会说，还能把事办了**。

这篇文章就按我一路追问、一路想通的顺序，用尽量生活化的方式把概念讲清楚，并把所有例子都换成——**旅游规划助手**：帮你安排出行、路线、吃住行、预算与注意事项。

---

## 0. 先给结论：Agent 不是一个模型，而是一套“把事办成”的系统

如果只用一句话解释：

> **Agent = LLM（大脑） + 工具（手脚） + 流程（规矩） + 记忆（记事本） + 监督（纠错/审计）**

所以“LLM 是智能体的大脑”没错，但**只有脑子不够**：  
你光会想，不会查、不敢订、不会校验，就做不出靠谱的旅行计划。

---

## 1. LLM 是大脑，但它不会“动手订票/查天气/比价格”

我后来真正想通的一点是：

- **LLM（大脑）**：负责理解需求、权衡方案、写计划、解释原因  
- **Agent/宿主程序（你写的后端）**：负责真实世界的动作，比如：
  - 查天气
  - 查航班/高铁时刻
  - 查景点开放时间
  - 查酒店价格与位置
  - 生成地图路线（或调用地图 API）
  - 下单/生成预订链接（如果你接了业务系统）

> 很像你找了一个“很会规划的朋友”。  
> 朋友能给建议，但真正去订票、去查实时信息，还是要靠工具和系统。

---

## 2. Function Calling：让 LLM 会“下指令”，程序去执行

**Function Calling（工具调用）**听起来高大上，其实就是：

> LLM 不直接干活，它只会“开工单”。  
> 程序看懂工单，然后去干活，再把结果带回来。

### 2.1 旅游例子（查天气）
你说：“我下周去成都，应该带什么衣服？”

LLM 可能会输出类似结构化指令：
- 工具名：`getWeatherForecast`
- 参数：`city=成都, dateRange=下周`

宿主程序真的去查天气 API，拿到结果比如：
- 12–20℃，多雨，早晚温差大

再把结果回填给 LLM，LLM 才能给出靠谱建议：
- 外套、雨具、防滑鞋、分层穿搭

一句话记住：
> **Function Calling 是“LLM 与程序之间的工单格式”。**

---

## 3. MCP：工具接入的“统一插头”

如果说 Function Calling 是“工单”，那 **MCP** 更像：

> “把各种外部工具接进来时，用统一规格的插头和插座。”

### 3.1 MCP 的直白解释
- **Agent 作为 MCP Client**：负责发请求
- **外部能力提供方是 MCP Server**：暴露工具/资源供调用

旅游助手里，可能的 MCP 工具包括：
- 地图/导航工具
- 天气工具
- 机酒搜索工具
- 汇率工具
- 翻译工具

> 关键边界：  
> **按 MCP 发请求的是 Agent/宿主程序，不是 LLM。**  
> LLM 仍只负责“决定用哪个工具、参数是什么”。

---

## 4. RAG：开卷考试，让模型“先翻资料再回答”

我对 RAG 最喜欢的比喻就是：

> **RAG 就是开卷考试。**

### 4.1 RAG（检索增强生成）是什么？（一句话）
> **RAG = 先从知识库里检索资料，再把资料塞给 LLM，让它“看着资料回答”。**

### 4.2 旅游助手为什么需要 RAG？
因为旅行规划里有大量“容易记错”的信息：
- 你自己整理的攻略/踩坑总结
- 公司/团队的差旅规定（如果是企业差旅助手）
- 你偏好的旅行风格（亲子/徒步/美食/省钱/舒适）
- 各景点的注意事项（来自你收集的可靠资料）

**旅游例子（避免胡编开放时间/玩法）**  
用户问：“我想去博物馆，周一开吗？”  
RAG 先从你自己的“景点信息文档/官方说明摘录”里找相关段落，再让 LLM回答，并附引用。

你得到的体验会更像：
> “助手拿着我准备的资料回答”，而不是“它凭感觉说”。

---

## 5. 上下文窗口：AI 的“短期记事本”，不是永久记忆

我曾经把“上下文窗口”理解成 AI 的记忆，后来发现更准确是：

> **上下文窗口 = 模型这一次推理能看到的输入内容上限。**

里面通常包括：
- 系统提示词（角色、规则）
- 历史对话
- RAG 检索片段
- 工具返回结果

它更像“桌面便签纸”，写得下就记得，写不下就要截断/总结。

真正的“长期记忆”（比如你喜欢住安静酒店、讨厌红眼航班、预算范围）通常要在外部存储里：
- 数据库
- 文档库
- 向量库（RAG）

需要时再取回来放进上下文里。

---

## 6. Skill：把“经验”封装成可复用技能（像旅行社 SOP）

你提出过一个非常准确的直觉：

> Skill 就是一系列能力的封装，让 AI 像人一样稳定、规范化执行。

### 6.1 Skill 是什么？（一句话）
> **Skill = 可复用的 SOP 模块：固定步骤 + 校验规则 + 输出模板。**

### 6.2 旅游例子：行程生成 Skill
一个“3 天游玩行程生成” Skill 可能包含：
1. 收集约束（天数、人数、偏好、预算、体力、出发地、到达时间）
2. 选区域（尽量少跨城折腾）
3. 排序景点（避免扎堆、避免闭馆日）
4. 计算通勤时间（太离谱就重排）
5. 输出结构化行程（每天：上午/下午/晚上、交通方式、预计耗时）
6. 给出备选方案（雨天/人多/临时闭园）

这就是“经验流程化”。

---

## 7. Workflow：把复杂事情“流程化”，让执行更可靠

我当时觉得它和 Skill 很像，后来区分清楚了：

### 7.1 Workflow 是什么？（一句话）
> **Workflow = 一条端到端流程，控制分支、重试、确认、人工介入。**

### 7.2 旅游例子：从需求到可执行计划的 Workflow
比如“帮我规划日本 7 天游”的流程可能是：
1. 收集信息（护照/签证、出发城市、预算、是否亲子）
2. 查天气与节假日（如果是樱花季/黄金周要提示风险）
3. 拉取景点/交通规则资料（RAG）
4. 生成路线初稿
5. 校验通勤时间与闭馆日
6. 需要用户确认关键偏好（例如“要不要环球影城？”）
7. 输出最终行程 + 预算表 + 行李清单 + 注意事项

一句话记住：
> **Skill 像招式，Workflow 像套路。**

---

## 8. Dify：把“搭旅游助手”这件事平台化

你问过：Dify 是不是定制工作流？

我们的结论是：
- **工作流是 Dify 的核心能力之一**
- 但 **Dify 不止工作流**，它是一个“搭建 LLM 应用/智能体的平台”：
  - 模型接入
  - 知识库（RAG）
  - 工作流/对话流编排
  - 工具/API 调用
  - 发布成 Web 应用或 API
  - 基础日志与运维

你可以用它很快做出“能跑的旅游助手原型”，再决定要不要深度代码化。

---

## 9. Spring AI / LangChain：在“写代码做 Agent”时帮你少造轮子

我最后也想明白了：
> 它们主要是 **LLM 应用开发底座**：统一接入模型，提供 RAG、工具调用、编排等积木。

- **Spring AI / Spring AI Alibaba**：更贴合企业 Java 工程化
- **LangChain / LangChain4j**：更偏“拼装”和快速试验

## 10. 一张最清晰的流程图：旅游助手是怎么“办事”的？

```mermaid
flowchart LR
  U[用户提问<br/>例如：帮我规划成都3日游] --> A[Agent/宿主程序<br/>(你写的后端)]
  A --> L[LLM<br/>(理解需求+做决策)]

  L -->|产生工具调用意图<br/>(Function Calling)| A

  A -->|检索攻略/规则/注意事项| R[RAG 知识库<br/>(pgvector)]
  R --> A

  A -->|查实时信息/执行动作| T[工具 Tools<br/>(天气/地图/交通/酒店)]
  T --> A

  A -->|可选：按标准协议接外部能力| M[MCP Server<br/>(地图/搜索等)]
  M --> A

  A -->|把证据/结果回填| L
  L --> A
  A --> U[输出给用户<br/>行程+预算+注意事项+引用]
```

## 11. 旅游助手里，每个名词的“生活例子速记”

- **LLM**：很会规划、很会解释的朋友
- **Agent**：这个朋友 + 手机/电脑 + 能查能订的工具 + 做事流程
- **Context Window**：朋友桌上的便签纸（这次能记住多少）
- **RAG**：把你收藏的攻略/官方说明翻出来给朋友看（开卷）
- **Tool**：查天气/查路线/查酒店的按钮
- **Function Calling**：朋友写“帮我查下周天气”的工单
- **Skill**：旅行社 SOP 模块（例如“生成3日行程模板”）
- **Workflow**：从收集需求到最终确认的完整流程
- **MCP**：接地图/搜索/酒店接口的统一插头
- **Dify**：搭旅游助手的工具箱平台（模型+RAG+流程+发布）

## 12. 我最关心的现实问题：初级 Java 怎么变成 Agent 工程师？

在我们对话中，你的条件是：
- 每天 2 小时
- 用云模型
- 技术栈：Spring AI Alibaba + PostgreSQL + pgvector
- 方向改成：**旅游规划助手**

你可以照这个目标做一个 8 周作品（核心不变，业务换成旅游）：
- **Week 1**：LLM 对话服务（稳定对话、日志清楚）
- **Week 2**：RAG（导入你整理的攻略/注意事项/规则，回答带引用）
- **Week 3**：工具调用（天气/地图路线/汇率/酒店查询的模拟或真实 API）
- **Week 4**：旅游助手 MVP（追问机制：城市/天数/预算/偏好；输出结构化行程）
- **Week 5**：Workflow/状态机（校验通勤、闭馆日、确认关键偏好）
- **Week 6**：权限/脱敏/审计（如果有用户信息/订单信息）
- **Week 7**：评测集 + 回归（60 条：行程、餐厅推荐、雨天备选等）
- **Week 8**：作品化发布（README、演示脚本、架构图）
