## 附录：特殊业务流程说明

### A. 信用分计算逻辑

```
基础规则：
- 每个行为产生一条 credit_log 记录
- 定期（如每天夜间）聚合结算，更新 credit_account

具体计算：

1. 打卡行为 (CHECKIN):
   base_score = sys_config.credit.checkin.base (默认2分)
   if media_count > 0:
       score += sys_config.credit.checkin.mediaBonus (默认1分)
   if content_length > 30:
       score += sys_config.credit.checkin.textBonus (默认1分)
   max_daily = sys_config.credit.checkin.dailyMax (默认1次)
   
   示例：有文字+图片的打卡 = 2 + 1 + 1 = 4分

2. 违规行为 (VIOLATION):
   penalty = sys_config.credit.violation.penalty (默认-20分)
   可根据违规严重程度调整倍数
   示例：虐待宠物 = -20分

3. 申请通过 (APPLICATION_APPROVED):
   已采纳的方案：在打卡时计分，申请本身不计分
   可选方案：申请通过时 +5分

4. 等级计算 (LEVEL):
   if score < 25: level = 0 (新注册)
   else if score < 50: level = 1 (铜牌)
   else if score < 100: level = 2 (银牌)
   else if score < 150: level = 3 (金牌)
   else: level = 4 (铂金)
```

### B. 推荐分计算逻辑

```
最终推荐分 = α·match_score + β·cf_score + γ·distance_score + δ·freshness_score

α = sys_config.recommend.alpha (默认0.55) - 偏好匹配权重
β = sys_config.recommend.beta (默认0.20) - 协同过滤权重
γ = sys_config.recommend.gamma (默认0.20) - 距离权重
δ = sys_config.recommend.delta (默认0.05) - 新鲜度权重

1. match_score (0-100):
   base = 0
   if pet.species == user.preference.species: base += 30
   if pet.size in user.preference.sizes: base += 15
   if pet.age in user.preference.ageRange: base += 15
   for tag in pet.tags:
       if tag in user.preference.tags: base += 8 (最多3个标签)
   if user.preference.requiredHealth:
       matched_count = count(pet健康特性 & user偏好)
       base += 10 * (matched_count / required_count)
   match_score = min(base, 100)

2. cf_score (0-100):
   根据用户行为相似度推荐得分
   - 相似用户收藏/申请过的宠物
   - 或相似宠物（与用户已交互宠物相似的）
   cf_score = 相似度 * 100

3. distance_score (0-20):
   if distance <= 5km: score = 20
   else if distance <= 20km: score = 20 * (1 - (distance-5)/15)
   else if distance <= maxDistance: score = 20 * (1 - distance/maxDistance)
   else: score = 0
   maxDistance = sys_config.recommend.maxDistanceKm (默认50)

4. freshness_score (0-10):
   days_since_publish = now - pet.publishedTime
   if days_since_publish <= 7: score = 10
   else if days_since_publish <= 30: score = 10 * (1 - (days-7)/23)
   else: score = 0
```

### C. 申请状态机

```
状态转移图：

SUBMITTED (用户提交)
    ↓ (机构审核)
UNDER_REVIEW (机构初审)
    ├→ REJECTED (拒绝)
    └→ INTERVIEW (约面谈)
         ├→ REJECTED (拒绝)
         └→ HOME_VISIT (家访，可选)
             ├→ REJECTED (拒绝)
             └→ APPROVED (通过)
                  ↓
                  宠物状态变更为 ADOPTED
                  其他申请自动变更为 REJECTED

CANCELLED (任何阶段可撤回)

关键业务规则：
- 同一宠物最多 N 个申请进行中（configurable）
- 申请通过后，宠物自动标记为 ADOPTED
- 其他该宠物的进行中申请自动拒绝（并通知）
- 每次状态变更记录流转日志
- 每次状态变更异步发送通知
```

### D. 宠物生命周期

```
生命周期状态：

DRAFT (草稿)
    ↓ (提交审核)
PENDING_AUDIT (待审核)
    ├→ REJECTED (驳回，返回DRAFT)
    └→ PUBLISHED (发布通过)
         ├→ APPLYING (有申请进行中，自动标记)
         └→ ADOPTED (领养通过)
         └→ REMOVED (下架/删除)

业务规则：
- DRAFT 状态机构可修改宠物信息
- PENDING_AUDIT 状态禁止修改
- PUBLISHED 状态有人申请时自动变更为 APPLYING
- ADOPTED 或 REMOVED 为终止状态
- 逻辑删除不真实删除数据（deleted=1）
```

### E. 打卡与回访规则

```
自动打卡提醒流程：

1. 申请通过（APPROVED）时：
   - 生成打卡任务：7天、30天、90天
   - 存储在定时任务表或消息队列

2. 定时任务执行（如凌晨2点）：
   - 扫描待提醒的打卡任务
   - 计算是否超期（距离adoptedTime的天数）
   - 若未打卡：
     if 超期3天内：level=warning, 发送提醒通知
     if 超期3天以上：level=alert, 发送催促通知 + 机构提醒
     if 超期30天：level=violation, 扣信用分（-5分），标记用户为风险

3. 机构回访：
   - 可查看领养人打卡情况
   - 若发现异常（无理由未打卡/虐待迹象）：
     → 标记黑名单（拒绝后续申请）
     → 上报管理员审查
     → 可触发违规扣分
```

### F. 权限边界检查清单

```
宠物操作权限：
✓ 机构只能修改/删除自己发布的宠物
✓ 管理员可查看所有宠物的审核状态
✓ 用户只能查看已发布的宠物

申请操作权限：
✓ 用户只能查看/撤回自己的申请
✓ 机构只能查看/处理自己宠物的申请
✓ 管理员无需审查申请（仅机构）

个人资料权限：
✓ 用户只能修改自己的资料和偏好
✓ 机构只能修改自己的机构资料
✓ 管理员可修改所有用户账户状态

管理员权限：
✓ 仅 role=ADMIN 的用户可访问管理接口
✓ 管理员操作需记录审计日志
```

## 附录：数据模型扩展说明

### 推荐结果缓存表（可选，用于性能优化）

若需要离线计算推荐结果，可创建：

```sql
CREATE TABLE recommend_result (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id BIGINT NOT NULL,
  pet_id BIGINT NOT NULL,
  score DECIMAL(5,2) NOT NULL,
  rank INT NOT NULL,
  reason VARCHAR(255),
  calc_time DATETIME NOT NULL,
  valid_until DATETIME NOT NULL,
  UNIQUE KEY uk_user_pet(user_id, pet_id),
  INDEX idx_user_valid(user_id, valid_until)
);
```

### 违规记录表（可选，用于黑名单管理）

```sql
CREATE TABLE violation_record (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id BIGINT NOT NULL,
  violation_type VARCHAR(32) NOT NULL COMMENT '类型：abuse/fraud/spam等',
  description VARCHAR(500),
  severity VARCHAR(16) NOT NULL COMMENT '严重程度：low/medium/high',
  status VARCHAR(16) NOT NULL DEFAULT 'PENDING' COMMENT '状态：pending/confirmed/dismissed',
  reported_by BIGINT,
  reported_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  confirmed_by BIGINT,
  confirmed_time DATETIME,
  penalty_applied TINYINT DEFAULT 0,
  penalty_score INT,
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_user_status(user_id, status),
  INDEX idx_reported_time(reported_time)
);
```

### 推荐训练日志表（可选）

```sql
CREATE TABLE recommend_training_log (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  training_date DATE NOT NULL,
  method VARCHAR(32) COMMENT '训练方法：item_cf/als/other',
  data_size INT,
  model_params JSON,
  performance JSON COMMENT '评估指标',
  status VARCHAR(16) COMMENT '状态：success/failed',
  error_msg VARCHAR(500),
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_date(training_date)
);
```
